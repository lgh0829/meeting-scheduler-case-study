# 모두에게 괜찮은 시간, 어떻게 찾을까

> 6명이 모이는 1시간짜리 회의. 누군가는 점심 직후를 피하고 싶고, 누군가는 특정 요일에 외근이 많고, 꼭 와야 하는 사람과 빠져도 되는 사람이 섞여 있습니다. 이 조건들을 반영해 회의 시간을 빠르게 확정하는 경험을 설계했습니다.

## 어떤 문제인가

회의 일정을 잡을 때 6명의 조건을 모두 같은 무게로 다룰 수는 없습니다. 가장 중요한 차이는 참석의 무게입니다. 꼭 참석해야 하는 사람이 있고, 빠져도 괜찮은 사람이 있습니다.

그런데 기존 일정 조율 도구에는 이 차이를 담을 수 없는 한계가 있습니다. 응답자 전원을 동일하게 취급해 **"전원 가능한 시간"만 찾다** 보니, 빠져도 괜찮은 한 명의 제약이 필수 참석자 전원이 가능한 현실적인 시간까지 지워버립니다. 결국 주최자는 스케줄링 도구 대신 메신저 도구를 사용하여 개별 조율에 많은 시간을 쏟게 됩니다.

## 어떻게 풀었나

기존 도구가 놓치던 정보, 참석의 무게를 스케줄링 도구가 수집하는 방식으로 해결했습니다. 여기에 참석자가 필수인지 선택인지, 안 되는 시간이 업무 일정 때문에 불가능한 시간인지 선호하지 않는 시간일 뿐인지를 구분해, 네 가지 조합을 각각 다르게 다룹니다.

| | 필수 참석자 | 선택 참석자 |
|---|---|---|
| **업무 일정** (외근, 다른 회의) | **하드 필터.** 겹치는 시간은 후보에서 제외합니다 | **1순위 일정 후보.** 참석 가능한 인원이 많을수록 후보가 위로 올라갑니다 |
| **선호 시간** (점심 직후 회피 등) | **1순위 일정 후보.** 회피 시간을 피한 후보가 위로 올라갑니다 | **2순위 일정 후보.** 후보가 비슷할 때 참고합니다 |

네 조합 중 후보를 걸러내는 기준은 필수 참석자의 업무 일정 하나뿐이고, 나머지 셋은 후보를 지우는 대신 순위를 매기는 신호로 쓰입니다. 후보를 거르는 기준이 좁아진 만큼 진행도 가벼워집니다. 선택 참석자가 응답하지 않아도 필수 참석자의 응답만 모이면 후보 탐색이 시작되고, 확정 통보도 무게에 맞춰 필수 참석자에게는 확정 사실을, 선택 참석자에게는 참석이 어려우면 빠져도 괜찮다는 메시지를 보냅니다.

그 결과 후보 시간은 "전원 가능한 시간" 하나로 좁혀지는 대신, 어떤 조건을 얼마나 만족하는지 비교할 수 있는 정보로 나타납니다. 마지막 선택은 주최자가 내립니다.

## 왜 이 방식으로 설계했나

"필수 참석자의 조건은 반드시 지켜져야 하고, 선택 참석자의 조건은 참고 정보로 충분하다."를 가설로 설정했습니다. 회의실 예약이나 반복 회의 같은 기능은 참석자간의 단발성으로 발생하는 회의 일정을 조율하는 문제와 직결되지 않아 처음부터 고려 범위에서 제외했습니다.

가설에 기반한 프로토타입의 UX는 다음과 같습니다. 필수/선택 참석자의 구분은 회의를 위한 조율 채널을 만들 시점에 주최자가 직접 설정합니다. 초대받은 참석자는 주간 시간표에서 자신의 일정을 주최자에게 전달합니다. 캘린더에서 자동으로 가져온 업무 일정 위에 피하고 싶은 시간만 직접 눌러 표시하면 되고, 선택 참석자는 응답을 건너뛰어도 필수 참석자의 응답만 모이면 조율은 다음 단계로 넘어갑니다. 참석자들의 일정이 모이면 주최자는 후보 시간 리스트에서 회의를 확정합니다. 각 후보에는 "필수 4/4 가능 · 회피 겹침 0건 · 선택 2/2 가능"처럼 판단 근거가 함께 표시되어, 주최자는 도구가 골라준 시간을 받아들이는 게 아니라 근거를 직접 보고 하나를 골라 확정합니다.
